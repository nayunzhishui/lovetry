# 情侣小程序开发指南

保存路径：`D:\codex\workspace\lovetry\情侣小程序开发指南.md`

本文档用于指导 `lovetry` 情侣小程序后续开发。它参考了 `D:\codex\workspace\safehome1.0` 的工作方法，也整理了 GitHub 上同类情侣小程序的可借鉴点。

## 1. 项目定位

本项目不是公开社交产品，而是给你和女朋友使用的私密情侣小程序。

建议定位：

> 一个只服务两个人的情侣空间，用来记录恋爱经历、复盘矛盾、保存纪念日、整理共同愿望，并帮助双方形成更清楚、更温和的沟通记录。

第一版优先目标：

- 两个人能进入同一个情侣空间；
- 能保存普通恋爱经历；
- 能保存吵架复盘；
- 能按时间线共同查看；
- 能沉淀下一步约定，而不是只记录情绪。

暂不建议第一版做：

- 复杂 AI 分析；
- 公开朋友圈；
- 付费系统；
- 多情侣/多用户社交；
- 复杂权限后台；
- 语音、视频、地图等高维护功能。

## 2. safehome 工作思路复用

`safehome1.0` 的核心不是具体技术栈，而是开发节奏：

```text
项目定位 -> MVP闭环 -> 最小数据流 -> 联调测试页 -> 正式页面 -> 文档交接 -> 小步迭代
```

迁移到本项目后：

```text
情侣使用场景 -> MVP闭环 -> 云函数/云数据库 -> 小程序调用层 -> 联调页 -> 正式页面 -> 版本验收
```

对应关系：

| safehome 做法 | lovetry 对应做法 |
|---|---|
| Flask 后端 API | 微信云函数 |
| SQLite 数据库 | 微信云数据库 |
| `backend/services` | `cloudfunctions` + `miniprogram/services` |
| `content/feedback_rules.json` | 沟通提示、复盘规则、纪念日文案 |
| `docs/API接口文档.md` | 云函数调用文档 |
| 联调测试页 | `pages/integration-test` |
| 非诊断、支持性反馈 | 非站队、非指责、促进沟通的复盘提示 |

本项目应该坚持：

- 每次只做一个功能；
- 每个功能先跑通数据流，再美化页面；
- 每次改动后写清楚运行方式和验收标准；
- 云函数、数据库字段、页面入口都要有文档；
- 先做你们两个人真的会用的功能，再考虑扩展。

- 吵架复盘要设计成“沟通工具”，不能做成“输赢/记仇工具”。

## 5. MVP 1.0 功能边界

MVP 1.0 核心闭环：

```text
创建情侣空间 -> 邀请另一半加入 -> 记录经历/复盘矛盾 -> 时间线查看 -> 写下下一步约定
```

必须完成：

1. 创建情侣空间；
2. 6 位加入码加入；
3. 恋爱经历记录；
4. 吵架复盘记录；
5. 时间线查看；
6. 云函数读写数据；
7. 基础空状态、加载状态、错误提示。

建议补齐：

1. 纪念日；
2. 心愿清单；
3. 记录详情页；
4. 编辑/删除记录；
5. 联调测试页；
6. 云函数调用封装。

暂缓：

1. 积分商城；
2. 宠物/亲密度；
3. 图片相册；
4. 订阅消息；
5. AI 总结；
6. 管理后台；
7. 正式发布审核。

### 6.1 推荐目录

当前项目已有基础目录，建议逐步补齐：

```text
couple-miniprogram/

      wishes/

    services/
      cloudApi.js
    utils/
      format.js
      validators.js
    app.js
    app.json
    app.wxss
    config.js
  cloudfunctions/
    login/
    couple/
    records/
    anniversaries/
    wishes/
```

### 6.2 页面职责

| 页面 | 职责 |
|---|---|
| 首页 | 显示情侣空间状态、最近记录、快捷入口 |
| 设置页 | 创建情侣空间、显示加入码、加入情侣空间 |
| 恋爱经历页 | 新增普通记录，后续可独立成 `moment-edit` |
| 吵架复盘页 | 按结构化字段保存复盘 |
| 时间线页 | 展示双方所有记录 |
| 记录详情页 | 查看完整内容和复盘字段 |
| 纪念日页 | 记录在一起日期、生日、自定义纪念日 |
| 心愿清单页 | 记录想一起完成的事情 |
| 联调测试页 | 验证 `login`、`couple`、`records` 数据流 |

### 6.3 前端原则

- 页面里少写重复的 `wx.cloud.callFunction`；
- 云函数调用集中到 `services/cloudApi.js`；
- 表单校验集中到 `utils/validators.js`；
- 时间格式集中到 `utils/format.js`；
- 每个页面都要有空状态、加载状态、失败提示；
- 所有保存按钮都要防重复点击；
- 记录提交成功后要清空表单或跳转详情页。

建议封装：

```js
// miniprogram/services/cloudApi.js
function call(name, data) {
  return wx.cloud.callFunction({ name, data }).then((res) => res.result);
}

module.exports = {
  login: () => call("login", {}),
  getMyCouple: () => call("couple", { action: "mine" }),
  createCouple: () => call("couple", { action: "create" }),
  joinCouple: (code) => call("couple", { action: "join", code }),
  createRecord: (record) => call("records", { action: "create", record }),
  listRecords: () => call("records", { action: "list" })
};
```

## 7. 后端开发指南

本项目后端优先使用微信云开发，不单独搭建服务器。

### 7.1 云函数职责

| 云函数 | 职责 |
|---|---|
| `login` | 获取 `openid` |
| `couple` | 创建、加入、查询情侣空间 |
| `records` | 创建、查询、编辑、删除记录 |
| `anniversaries` | 管理纪念日 |
| `wishes` | 管理心愿清单 |
| `stats` | 后续统计记录数量、复盘趋势 |

### 7.2 数据库集合

建议集合：

```text
couples
records
anniversaries
wishes
users
```

`couples`：

```js
{
  _id: "情侣空间ID",
  code: "6位加入码",
  members: ["openid1", "openid2"],
  createdBy: "openid1",
  createdAt: Date,
  updatedAt: Date
}
```

`records`：

```js
{
  _id: "记录ID",
  coupleId: "情侣空间ID",
  type: "moment | conflict",
  title: "标题",
  content: "正文",
  payload: {
    conflictTime: "争吵时间",
    stressEvents: "外部压力",
    bodyState: "身体状态",
    sleepState: "睡眠状态",
    relationshipState: "关系状态",
    communication: "沟通过程",
    satisfaction: 1,
    result: "下一步约定"
  },
  creatorOpenid: "openid",
  visibility: "couple",
  createdAt: Date,
  updatedAt: Date,
  deletedAt: null
}
```

`anniversaries`：

```js
{
  coupleId: "情侣空间ID",
  title: "在一起纪念日",
  date: "2026-05-15",
  type: "together | birthday | custom",
  createdBy: "openid",
  createdAt: Date,
  updatedAt: Date
}
```

`wishes`：

```js
{
  coupleId: "情侣空间ID",
  title: "一起去旅行",
  detail: "说明",
  status: "todo | doing | done",
  createdBy: "openid",
  completedBy: "openid",
  completedAt: null,
  createdAt: Date,
  updatedAt: Date
}
```

### 7.3 后端安全原则

- 数据读写必须通过云函数；
- 小程序端不直接开放数据库写权限；
- 查询记录时必须先确认当前 `OPENID` 属于该 `coupleId`；
- 加入码只允许两人加入；
- 删除优先软删除，保留 `deletedAt`；
- 云函数要返回明确错误码，前端再转成人能看懂的提示；
- 不在日志里输出完整隐私内容。

### 7.4 云函数错误码

建议统一：

```text
COUPLE_REQUIRED      尚未创建或加入情侣空间
JOIN_CODE_REQUIRED   缺少加入码
COUPLE_NOT_FOUND     未找到情侣空间
COUPLE_FULL          情侣空间已满
INVALID_RECORD       记录字段不完整
NO_PERMISSION        无权限访问
UNKNOWN_ACTION       未知操作
```

## 8. 小程序 UI 指南

### 8.1 视觉方向

建议风格：

- 温暖；
- 私密；
- 干净；
- 不幼稚；
- 不做大面积粉色；
- 不做廉价恋爱模板感。

推荐色彩：

```text
背景：#F7F4EF / #FFFDFC
主色：#D96C75
强调：#8A7CCF
文字：#2E2A27
弱文字：#7E746C
线框：#E9DFD6
成功：#4E9F6D
警示：#D98C44
```

### 8.2 UI 结构

首页优先显示：

1. 当前情侣空间；
2. 在一起天数或最近纪念日；
3. 最近一条记录；
4. 三个主入口：记录经历、吵架复盘、时间线；
5. 后续入口：纪念日、心愿清单。

不要把首页做成宣传页。打开小程序第一屏就应该能使用。

### 8.3 表单体验

吵架复盘页建议按区块拆：

```text
发生了什么
我当时的状态
对方可能的状态
我们怎么沟通的
这次沟通满意度
下次想怎么做
```

文案要避免站队：

- 不写“谁错了”；
- 不写“对方的问题”；
- 不写“你赢了吗”；
- 改成“事实是什么”“我感受到什么”“我需要什么”“下一步怎么约定”。

### 8.4 组件建议

MVP 可先用原生组件。等功能稳定后再考虑：

- [TDesign Miniprogram](https://github.com/Tencent/tdesign-miniprogram)
- [Vant Weapp](https://github.com/youzan/vant-weapp)

第一版不建议立即引入组件库，避免增加 npm 构建和样式迁移成本。

## 10. 验收标准

MVP 1.0 验收：

- 微信开发者工具能打开项目；
- 云开发环境 ID 已配置；
- `login`、`couple`、`records` 已上传部署；
- 数据库有 `couples`、`records` 集合；
- 用户 A 能创建情侣空间；
- 用户 B 能用加入码加入；
- 双方能写恋爱经历；
- 双方能写吵架复盘；
- 时间线只显示当前情侣空间的数据；
- 无情侣空间时页面有清楚提示；
- 保存失败时有清楚提示；
- 数据库权限不直接暴露给小程序端。
