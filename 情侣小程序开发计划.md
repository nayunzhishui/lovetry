## 1. 项目目标

开发一个仅供一对情侣使用的微信小程序，用于共同记录、规划和回顾生活。核心原则：

- 双人数据同步可靠；
- 情侣空间之间严格隔离；
- 敏感记录默认最小可见；
- 吵架记录帮助复盘，不判断谁对谁错；
- 先完成稳定闭环，再扩展媒体和统计；
- 保持微信原生小程序 + CloudBase 技术路线，不引入独立服务器；
- 每个小任务都包含代码、验证、文档和可交接状态。

### 1.1 MVP 核心闭环

```text
创建情侣空间 → 邀请伴侣加入 → 共同记录/共同计划
→ 日历聚合 → 奖励互动 → 回顾统计 → 数据备份
```

### 1.2 当前事实基准

当前已有：

- 微信原生小程序骨架；
- `login`、`couple`、`records` 三个云函数；
- 首页、吵架复盘、时间线、情侣设置四个页面；
- 情侣空间创建、加入码加入、普通记录、吵架记录和时间线雏形。

当前不足：

- `AppID` 为 `touristappid`，云环境 ID 为占位值；
- 页面直接调用云函数，缺少统一调用模块；
- 缺少统一结果、错误码、加载态和防重复提交；
- 缺少编辑、软删除、可见性、并发版本与双账号验收；
- 加入码使用 `Math.random()`，缺少唯一性、过期和并发加入保护；
- 缺少自动测试、联调页、真实开发者工具和真机验收；
- 原 `项目逐步开发清单.md` 含重复菜单、双重日历、末期隐私审查等问题，不再作为唯一执行入口。

## 4. 技术方案

### 4.1 总体架构

```text
小程序页面
  ↓
services/cloudApi.js（唯一前端调用接口）
  ↓
login / couple / records / plans / rewards / media / dashboard
  ↓
CloudBase 云数据库 + 云存储
```

不迁移到 uni-app、Taro、Spring Boot 或 SafeHome 的 Flask 后端。复用 SafeHome 的事实基准、执行入口、联调页、验收证据和小步交接方法。

### 4.2 深模块与接口

五个业务模块：

1. 情侣空间模块：身份、情侣空间、加入、成员权限；
2. 共同记录模块：日记、吵架、睡眠、生理期、玩乐、游戏、番茄钟；
3. 共同计划模块：任务、事件、菜单、旅行、纪念日；
4. 奖励模块：钱包、流水、兑换和任务奖励；
5. 媒体模块：相册、图片元数据、访问和删除。

首页、日历和统计属于读取聚合，不拥有另一套业务数据。

### 4.3 云函数

```text
login       保留：获取当前 OpenID
couple      扩展：创建、加入、查询、更新资料、退出
records     扩展：记录 CRUD、筛选、详情、统计
plans       新增：任务、事件、菜单、旅行、纪念日
rewards     新增：钱包、流水、奖励、消费、撤销
media       新增：相册和图片元数据
dashboard   新增：首页、日历和统计聚合
```

统一返回：

```js
{ ok: true, data: {} }
{ ok: false, error: { code: "NO_PERMISSION", message: "无权访问" } }
```

### 4.4 数据集合

```text
couples
memberships
records
plans
wallets
reward_transactions
albums
media_assets
```

记录公共字段：

```js
{
  coupleId,
  type: "moment | mood | conflict | outing | sleep | period | game | pomodoro",
  visibility: "private | couple",
  ownerOpenid,
  title,
  content,
  startAt,
  endAt,
  metrics: {},
  relatedPlanId,
  version,
  createdAt,
  updatedAt,
  deletedAt
}
```

计划公共字段：

```js
{
  coupleId,
  type: "task | event | menu | trip | anniversary",
  title,
  detail,
  status: "todo | doing | done | archived",
  assigneeOpenids: [],
  startAt,
  endAt,
  rewardPoints,
  payload: {},
  version,
  createdBy,
  createdAt,
  updatedAt,
  deletedAt
}
```

奖励流水必须包含 `sourceType + sourceId + idempotencyKey`，任务重复完成不得重复发放奖励。

### 4.5 隐私与安全

- 所有数据库读写经云函数；
- 每次查询、详情、修改、删除都校验情侣成员关系；
- 生理期、睡眠、番茄钟默认私密；
- 心情和吵架记录由创建者选择是否共享；
- 删除采用软删除，图片删除同时清理云存储；
- 加入码使用安全随机数、唯一检查、过期时间、失败次数限制；
- 第二位成员加入使用事务或条件更新，避免第三人并发加入；
- 奖励发放、任务完成、钱包余额和流水写入使用服务端事务；
- 不在日志输出记录正文、生理期内容、图片地址等隐私；
- 支持导出自己的数据和注销/解除情侣关系流程。

### 4.6 页面信息架构

```text
首页：在一起天数、今日任务、最近心情、余额、快捷新增
日历：月历、事件点、类型筛选、日期详情
记录：心情、睡眠、生理期、吵架、玩乐、游戏、番茄钟
我们：相册、旅行、菜单、奖励、情侣设置
```

视觉方向：温暖、私密、克制、非模板化。签名元素为“两个人的双轨时间线”：共享事件显示为两条细线汇合，私密记录只显示本人单轨。建立统一颜色、字体、间距、圆角和阴影变量，避免大面积粉色与无意义装饰。

## 5. 自动化执行规则

- 严格按依赖顺序执行；
- 每次只把一个小任务从未开始推进到完成；
- 业务规则优先采用测试驱动：一个行为测试 → 最小实现 → 通过 → 下一个行为；
- 新页面同步注册 `app.json`；
- 新云函数必须包含 `index.js`、`package.json`；
- 页面必须具备加载、空、失败、防重复提交状态；
- 每完成一个小任务，更新 `项目执行记录.md` 和本文件复选框；
- 不把模拟器通过写成真机通过；人工外部验收必须单独标记；
- 不删除现有可用能力，字段迁移需兼容旧数据；
- 提交前执行 `git status`、静态检查、自动测试和差异检查；
- Git 提交采用 `type(scope): summary`，一个提交对应一个可审查闭环。
