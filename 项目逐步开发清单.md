### 0.1 执行总规则

所有后续开发必须遵守：

- 每轮只执行一个任务编号，例如 `T00` 或 `T06`。
- 不允许一次执行多个功能模块，除非用户明确要求。
- 新增页面必须同步注册到 `miniprogram/app.json`。
- 新增云函数必须包含 `index.js` 和 `package.json`。
- 新增云函数 action 必须写清入参、返回值和错误码。
- 新增集合必须同步本文档和后续 `docs/数据库字段说明.md`。
- 新增页面必须包含加载态、空状态、失败提示和保存防重复点击。
- 敏感数据默认最小可见，尤其是生理期、心情、睡眠、吵架复盘。
- 不允许前端页面直接读写云数据库，必须经过云函数。
- 不删除现有 `login`、`couple`、`records` 云函数。
- 不删除现有 `pages/index`、`pages/conflict`、`pages/timeline`、`pages/settings`。
- 每轮结束必须写清：改动文件、验证结果、未完成事项、下一步建议。

调研结论：

- 情侣类项目常见基础是“情侣绑定 + 云同步 + 日常动态/相册 + 心愿/任务”；
- 任务积分系统必须有流水，不能只保存一个余额数字；
- 日历适合做所有模块的统一入口，但不要第一步就引入复杂组件；
- 相册会涉及云存储费用、权限、删除和 fileID 转换，建议后置；
- 生理期、睡眠属于敏感数据，要支持“仅自己可见/情侣可见”。

## 5. 通用后端规范

### 5.1 云函数返回格式

所有新云函数统一返回：

```js
{
  ok: true,
  data: {},
  error: null
}
```

失败返回：

```js
{
  ok: false,
  data: null,
  error: {
    code: "COUPLE_REQUIRED",
    message: "请先创建或加入情侣空间"
  }
}
```

### 5.2 通用字段

所有业务集合统一字段：

```js
{
  coupleId: "情侣空间ID",
  creatorOpenid: "创建者openid",
  visibility: "couple | private",
  createdAt: Date,
  updatedAt: Date,
  deletedAt: null,
  version: 1
}
```

### 5.3 通用权限

所有云函数读写前必须检查：

1. 当前用户是否有 `OPENID`；
2. 当前用户是否加入了情侣空间；
3. 目标数据的 `coupleId` 是否属于当前情侣空间；
4. `visibility === "private"` 时，只允许创建者读取；
5. 删除使用软删除，写入 `deletedAt`。

### 5.4 通用 action

每个业务云函数优先支持：

```text
create
update
delete
detail
list
listByDate
```

统计类额外支持：

```text
summary
stats
```

## 14. 风险与注意事项

### 14.1 云开发费用

相册和图片会使用云存储，可能产生费用。建议：

- 先限制单次上传 3 张；
- 限制图片大小；
- 后续再做压缩；
- 删除图片时同步删除云存储文件。

### 14.2 隐私

生理期、心情、吵架记录可能敏感。建议：

- 默认私密的模块：生理期；
- 可选择私密的模块：心情、睡眠、吵架；
- 默认情侣可见的模块：任务、旅行、奖励、日历手动事件。

### 14.3 关系体验

吵架记录不要做成“记仇工具”。文案应强调：

- 事实；
- 感受；
- 需求；
- 责任；
- 下一步约定。

避免：

- 谁赢了；
- 谁错了；
- 给对方贴标签；
- 自动判断分手/不合适。

### 14.4 数据一致性

奖励账户、任务完成、番茄钟奖励之间有关联，必须注意：

- 奖励流水只写一次；
- 重复点击完成不能重复发奖励；
- `sourceType + sourceId` 应唯一；
